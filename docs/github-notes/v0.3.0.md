# Release v0.3.0 — Connectors + Agent Exporters + Feedback Loop + Enterprise Docs

> Direct API access, export to any AI agent, close the verification loop, and enterprise-ready documentation.

This release completes Phase 2 of intake's roadmap: 3 API connectors for pulling requirements directly from Jira, Confluence, and GitHub; 4 new agent-specific exporters (6 total); a feedback loop that uses LLM analysis to explain verification failures and suggest spec amendments; and comprehensive enterprise documentation covering security, deployment, CI/CD integration, and workflow patterns.

## Install

```bash
pip install intake-ai-cli

# With connector dependencies (Jira, Confluence, GitHub)
pip install "intake-ai-cli[connectors]"
```

Requires Python 3.12+.

## What's New

### 3 API Connectors

Pull requirements directly from issue trackers and wikis without manual export:

```bash
# Jira: single issue, multiple, JQL query, sprint
intake init "Sprint tasks" -s jira://PROJ/sprint/42
intake init "P1 bugs" -s "jira://PROJ?jql=priority=Highest"
intake init "Feature" -s jira://PROJ-123,PROJ-124,PROJ-125

# Confluence: page by ID, space/title, CQL search
intake init "RFC review" -s confluence://ENG/Architecture-RFC
intake init "Docs" -s "confluence://?cql=space=ENG AND type=page"

# GitHub: single/multiple issues, filtered queries
intake init "Bug triage" -s "github://org/repo/issues?labels=bug&state=open"
intake init "Feature" -s github://org/repo/issues/42
```

Connectors use async fetch, save results to temp files, and feed into existing parsers (Jira JSON, Confluence HTML, GitHub Issues JSON). Requires optional dependencies: `pip install "intake-ai-cli[connectors]"`.

#### Connector Configuration

```yaml
# .intake.yaml
connectors:
  jira:
    url: "https://company.atlassian.net"
    auth_type: token
    token_env: JIRA_API_TOKEN
    email_env: JIRA_EMAIL
    include_comments: true
    max_comments: 5
    default_project: PROJ
    fields:
      - summary
      - description
      - priority
      - status
      - labels
  confluence:
    url: "https://company.atlassian.net/wiki"
    token_env: CONFLUENCE_API_TOKEN
    email_env: CONFLUENCE_EMAIL
    include_child_pages: false
    max_depth: 2
  github:
    token_env: GITHUB_TOKEN
    default_repo: "org/repo"
```

#### Connector Modules

| Module | Description |
|--------|-------------|
| `connectors/base.py` | ConnectorRegistry, ConnectorError, ConnectorNotFoundError |
| `connectors/jira_api.py` | Jira REST API — issues, JQL, sprints. Lazy import of `atlassian-python-api` |
| `connectors/confluence_api.py` | Confluence REST API — pages, spaces, CQL. Outputs HTML for ConfluenceParser |
| `connectors/github_api.py` | GitHub API via PyGithub — issues, labels, filters. Max 50 issues, 10 comments |

### 4 New Exporters (6 Total)

Export specs to the native format of any major AI coding agent:

```bash
intake export specs/my-feature/ -f claude-code -o .
intake export specs/my-feature/ -f cursor -o .
intake export specs/my-feature/ -f kiro -o .
intake export specs/my-feature/ -f copilot -o .
```

| Exporter | Files Generated | Target Agent |
|----------|----------------|--------------|
| **Claude Code** | `CLAUDE.md` + `.intake/tasks/TASK-NNN.md` + `.intake/verify.sh` + `.intake/spec-summary.md` | Claude Code |
| **Cursor** | `.cursor/rules/intake-spec.mdc` (YAML frontmatter + Markdown) | Cursor |
| **Kiro** | `requirements.md` + `design.md` + `tasks.md` (native Kiro format with checkboxes) | Kiro |
| **Copilot** | `.github/copilot-instructions.md` | GitHub Copilot |
| architect | `pipeline.yaml` + `spec/` copy | Any orchestrator |
| generic | `SPEC.md` + `verify.sh` + `spec/` copy | Any agent |

All 4 new exporters implement the **V2 ExporterPlugin protocol**:
- `meta` — plugin metadata (name, version, description)
- `supported_agents` — list of compatible agents
- `export()` → `ExportResult` — dataclass with generated files, locations, and metadata

Claude Code exporter intelligently handles existing `CLAUDE.md` files: if one exists, it appends the spec section (or replaces an existing intake section) without overwriting user content.

### Feedback Loop

Closes the verification-implementation cycle. When `intake verify` reports failures, `intake feedback` uses an LLM to analyze what went wrong and suggest fixes:

```bash
# Analyze verification failures
intake feedback specs/my-feature/ -p .

# Auto-apply spec amendments (preview first, then apply)
intake feedback specs/my-feature/ -p . --apply

# Format suggestions for a specific agent
intake feedback specs/my-feature/ -p . --agent-format claude-code
intake feedback specs/my-feature/ -p . --agent-format cursor
```

For each failed check, the feedback module identifies:

| Field | Description |
|-------|-------------|
| Root cause | Why the check failed |
| Suggestion | Concrete fix with file paths and code hints |
| Severity | `critical`, `major`, or `minor` |
| Category | `implementation_gap`, `spec_ambiguity`, `config_issue`, `dependency_missing`, `test_issue` |
| Affected tasks | Which spec tasks relate to the failure |
| Spec amendment | Proposed change to the spec (add/modify/remove sections) |

#### Feedback Modules

| Module | Purpose |
|--------|---------|
| `feedback/analyzer.py` | LLM-based failure analysis. Dataclasses: SpecAmendment, FailureAnalysis, FeedbackResult |
| `feedback/prompts.py` | Analysis prompt with `{language}` placeholder, structured JSON output schema |
| `feedback/suggestions.py` | SuggestionFormatter — terminal output (Rich) + agent-specific formatting (Jinja2) |
| `feedback/spec_updater.py` | SpecUpdater — `preview()` and `apply()` spec amendments via regex section matching |

#### Feedback Configuration

```yaml
# .intake.yaml
feedback:
  auto_amend_spec: false        # If true, --apply is default
  max_suggestions: 10           # Limit suggestions per analysis
  include_code_snippets: true   # Include code context in suggestions
```

### Export Helpers

New shared module `export/_helpers.py` provides utilities used by all V2 exporters:

| Function | Purpose |
|----------|---------|
| `read_spec_file()` | Safe file reading with encoding fallback |
| `parse_tasks()` | Extract task list from tasks.md (id, title, status, deps) |
| `load_acceptance_checks()` | Parse acceptance.yaml into check objects |
| `summarize_content()` | Truncate content for space-constrained formats |
| `count_requirements()` | Count FR/NFR from requirements.md |

| Code | Meaning |

## Migration Guide

No migration needed. All existing specs, configurations, and workflows continue to work unchanged.

To use the new features:

```bash
# Upgrade
pip install --upgrade intake-ai-cli

# Install connector dependencies (optional)
pip install "intake-ai-cli[connectors]"

# Verify
intake --version     # 0.3.0
intake plugins list  # shows 20 plugins (11 parsers + 6 exporters + 3 connectors)

# Try API connectors
export JIRA_API_TOKEN=your-token
export JIRA_EMAIL=you@company.com
intake init "Sprint" -s jira://PROJ/sprint/42

# Try new exporters
intake export specs/my-feature/ -f claude-code -o .
intake export specs/my-feature/ -f cursor -o .
intake export specs/my-feature/ -f kiro -o .
intake export specs/my-feature/ -f copilot -o .

# Try feedback loop
intake verify specs/my-feature/ -p .
intake feedback specs/my-feature/ -p .
intake feedback specs/my-feature/ -p . --apply --agent-format claude-code
```
