# Release v1.0.0 — Production Ready

> QA hardening, `intake regenerate` command, 12 bug fixes, 902 tests passing.

intake v1.0.0 is the first production-ready release. This version focuses on quality: 12 bugs were identified and fixed during a comprehensive QA audit, a new `intake regenerate` command was added, and the test suite grew to 902 tests with zero mypy/ruff errors.

## Install

```bash
pip install intake-ai-cli

# With all connector dependencies (Jira, Confluence, GitHub, GitLab)
pip install "intake-ai-cli[connectors]"

# Everything
pip install "intake-ai-cli[all]"
```

Requires Python 3.12+.

## Highlights

- **`intake regenerate` command** — regenerate a spec from scratch, re-analyzing all sources
- **12 bug fixes** — from critical YAML injection to improved doctor diagnostics
- **acceptance.yaml now uses `yaml.dump()`** — guaranteed valid YAML regardless of LLM content
- **Config loader uses `model_validate()`** — correct deep deserialization of nested Pydantic models
- **Doctor validates Pydantic schema** — catches invalid config values, not just YAML syntax
- **Doctor detects custom API keys** — reads `llm.api_key_env` from `.intake.yaml`
- **LLM check type normalization** — invalid types like "grep" or "manual" are auto-mapped
- **`intake list` recursive search** — finds specs in subdirectories
- **`verify --format json` clean output** — structlog routed to stderr, stdout is pure JSON
- **902 tests**, **0 mypy --strict errors**, **0 ruff errors**

## What's New

### `intake regenerate` Command

Regenerate a spec completely from its original sources:

```bash
# Re-analyze and regenerate all 6 spec files
intake regenerate specs/my-feature/ -s requirements.md -s notes.md

# With custom model
intake regenerate specs/my-feature/ -s reqs.md --model gpt-4o
```

This is equivalent to `intake add SPEC_DIR --regenerate -s SOURCES` but with a clearer UX. Useful when sources have changed significantly and incremental `add` isn't sufficient.

## Bug Fixes

### Critical

| ID | Component | Issue | Fix |
|----|-----------|-------|-----|
| BUG-006 | `config/loader.py` | `model_copy(update=...)` left nested Pydantic models as raw dicts | Changed `_merge_yaml()` and `_merge_overrides()` to use `model_validate()` |
| BUG-008 | `generate/spec_builder.py` | Jinja2 template produced invalid YAML when commands contained quotes | Replaced template with programmatic `yaml.dump()` |

### High Priority

| ID | Component | Issue | Fix |
|----|-----------|-------|-----|
| BUG-001 | `doctor/checks.py` | Custom API key env vars (e.g., LITELLM_API_KEY) not detected | `_check_api_key()` now reads `llm.api_key_env` from `.intake.yaml` |
| BUG-009 | `analyze/design.py` | LLMs return invalid check types ("grep", "code_pattern", "manual") | Added `_normalize_check_type()` with alias map + fallback to "command" |

### Medium Priority

| ID | Component | Issue | Fix |
|----|-----------|-------|-----|
| BUG-003 | `estimate/estimator.py` | `estimate_from_files()` silently returned zero for missing files | Now raises `FileNotFoundError` with list of missing files |
| BUG-004 | `cli.py` | `init -o path/name` used entire path as output_dir | Parent is `output_dir`, basename is `spec_name` |
| BUG-010 | `cli.py` | `intake list` only found specs in top-level directory | Changed `iterdir()` to `rglob()` for recursive search |
| BUG-012 | `cli.py` | `verify --format json` mixed structlog output into stdout | `setup_logging(verbose=False)` routes logs to stderr; uses `click.echo()` for JSON |

### Low Priority

| ID | Component | Issue | Fix |
|----|-----------|-------|-----|
| BUG-002 | `doctor/checks.py` | Config check only validated YAML syntax, not schema | Now validates against full `IntakeConfig` Pydantic schema |
| BUG-005 | `analyze/design.py` | Design prompt didn't list valid check types | Added explicit valid type list to prompt |
| BUG-007 | `validate/checker.py` | Cross-reference check didn't validate acceptance→requirement refs | Now checks that acceptance check names/tags match existing requirement IDs |
| BUG-011 | `generate/spec_builder.py` | `acceptance.yaml.j2` template still shipped but unused | Kept for backward compatibility; generation uses `yaml.dump()` |

## Architecture Changes

### Modified Modules

| Module | Change |
|--------|--------|
| `cli.py` | New `regenerate` command, fixed `init -o`, recursive `list`, clean JSON `verify` |
| `config/loader.py` | `model_validate()` for nested model deserialization |
| `generate/spec_builder.py` | `yaml.dump()` replaces Jinja2 for acceptance.yaml |
| `analyze/design.py` | `_normalize_check_type()` + valid types in prompt |
| `doctor/checks.py` | Custom API key detection, Pydantic schema validation |
| `estimate/estimator.py` | `FileNotFoundError` for missing files |
| `validate/checker.py` | Acceptance→requirement cross-reference checks |

### Technical Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 47 | `yaml.dump()` over Jinja2 for YAML | Eliminates entire class of quote/escaping bugs |
| 48 | `model_validate()` over `model_copy(update=...)` | Correct deep deserialization of nested Pydantic v2 models |
| 49 | Alias map for check types | LLMs are unpredictable; normalize at ingestion rather than reject |
| 50 | `rglob("spec.lock.yaml")` for list | Natural behavior — users expect recursive search |
| 51 | `click.echo()` for machine-readable output | Rich console adds formatting that breaks JSON/JUnit parsing |

## Key Lessons Learned

1. **Never use `model_copy(update=...)` for Pydantic v2 nested models** — use `model_validate()` instead. `model_copy` does shallow updates, leaving nested models as raw dicts.
2. **Never use Jinja2 to generate YAML with dynamic content** — use `yaml.dump()`. Template-based YAML generation is fragile with special characters.
3. **LLMs invent check types** — always normalize with an alias map + fallback rather than rejecting invalid types.
4. **structlog must go to stderr** — any tool that outputs structured data (JSON, JUnit) to stdout will break if logs are mixed in.
