# Release v0.1.0 — Initial Release

> From requirements in any format to verified implementation.

**intake** is an open-source CLI tool that transforms requirements from any format — Jira exports, Confluence pages, PDFs, Markdown, YAML, images, DOCX, free text — into a normalized, verifiable spec that any AI coding agent can consume.

## Install

```bash
pip install intake-ai-cli
```

Requires Python 3.12+.

---

## Highlights

- **8 input parsers** with automatic format detection
- **LLM-powered analysis** with extraction, deduplication, conflict detection, risk assessment, and technical design
- **6 spec files** generated per feature: requirements, design, tasks, acceptance checks, context, and source traceability
- **Verification engine** with 4 check types and 3 report formats (terminal, JSON, JUnit XML)
- **2 export formats**: architect (pipeline.yaml) and generic (SPEC.md + verify.sh)
- **Provider-agnostic**: works with any model LiteLLM supports (Anthropic, OpenAI, Google, local models)
- **Budget enforcement**: configurable cost limits per spec with automatic tracking
- **313 tests**, **83% coverage**, **0 ruff errors**, **0 mypy --strict errors**

## What's Included

### CLI — 8 Commands

| Command | Description |
|---------|-------------|
| `intake init` | Generate a complete spec from requirement sources (full pipeline) |
| `intake add` | Add sources to an existing spec incrementally |
| `intake verify` | Run acceptance checks against an implementation |
| `intake export` | Export a spec to an agent-ready format |
| `intake show` | Display spec summary (requirements, tasks, cost, model) |
| `intake list` | List all specs in a directory |
| `intake diff` | Compare two spec versions (added/removed/modified) |
| `intake doctor` | Check environment health with optional `--fix` auto-repair |

### Phase 1 — Ingest (8 Parsers)

| Parser | Formats | What It Extracts |
|--------|---------|------------------|
| **Markdown** | `.md`, `.markdown` | YAML front matter, heading-based sections |
| **Plaintext** | `.txt`, stdin (`-`) | Paragraph-level sections, Slack thread dumps |
| **YAML/JSON** | `.yaml`, `.yml`, `.json` | Top-level keys as sections, structured data |
| **PDF** | `.pdf` | Page text + tables (via pdfplumber) |
| **DOCX** | `.docx` | Paragraphs, tables, headings, document metadata (via python-docx) |
| **Jira** | `.json` (auto-detected) | Issues, comments (last 5), labels, priority, status, inter-issue links, ADF support |
| **Confluence** | `.html` (auto-detected) | Clean Markdown from HTML, headings, metadata (via BS4 + markdownify) |
| **Image** | `.png`, `.jpg`, `.webp`, `.gif` | Visual content description via LLM vision |

All parsers produce normalized `ParsedContent` with text, metadata, sections, and relations.

**Auto-detection**: Format is detected by file extension and content inspection. Jira JSON and Confluence HTML are distinguished automatically from generic JSON/HTML files.

### Phase 2 — Analyze (LLM Pipeline)

The async analysis pipeline coordinates 6 sub-phases:

1. **Extraction** — LLM extracts functional requirements (FR-XX), non-functional requirements (NFR-XX), conflicts, and open questions
2. **Deduplication** — Jaccard word similarity (threshold 0.75) removes duplicate requirements across sources
3. **Conflict validation** — Filters incomplete conflict entries
4. **Question validation** — Filters incomplete open questions
5. **Risk assessment** (optional) — LLM evaluates risks per requirement with probability, impact, and mitigation
6. **Design** — LLM produces architecture components, file actions, implementation tasks (DAG), and executable acceptance checks

**10 typed data models**: Requirement, Conflict, OpenQuestion, RiskItem, TechDecision, TaskItem, FileAction, AcceptanceCheck, DesignResult, AnalysisResult.

### Phase 3 — Generate (6 Spec Files + Lock)

| File | Content |
|------|---------|
| `requirements.md` | Functional & non-functional requirements, conflicts, open questions |
| `design.md` | Architecture components, file actions, technical decisions, dependencies |
| `tasks.md` | Implementation tasks with dependencies, estimates, and verification checks |
| `acceptance.yaml` | Executable acceptance checks (command, files_exist, pattern_present, pattern_absent) |
| `context.md` | Project context for AI agents: stack, conventions, risk summary |
| `sources.md` | Full traceability: every requirement mapped to its original source |
| `spec.lock.yaml` | Reproducibility lock with SHA-256 hashes, cost, and timestamps |

Templates are rendered with Jinja2. The lock file enables staleness detection.

### Phase 4 — Verify (Acceptance Engine)

4 check types:

| Type | Behavior |
|------|----------|
| `command` | Run a shell command, check exit code == 0 |
| `files_exist` | Verify listed file paths exist |
| `pattern_present` | Verify regex patterns exist in files matching a glob |
| `pattern_absent` | Verify regex patterns do NOT exist in files matching a glob |

Features: tag-based filtering (`--tags`), fail-fast mode, configurable timeout per check, required vs optional checks.

3 report formats:

| Format | Output |
|--------|--------|
| `terminal` | Rich table with colors |
| `json` | Machine-readable JSON |
| `junit` | JUnit XML for CI (GitHub Actions, Jenkins) |

### Phase 5 — Export (Agent-Ready Output)

| Format | Generated Files |
|--------|----------------|
| **architect** | `pipeline.yaml` (one step per task + final verification) + spec copy |
| **generic** | `SPEC.md` (consolidated Markdown) + `verify.sh` (standalone bash script) + spec copy |

The `verify.sh` script runs all command checks without requiring intake to be installed.

### Configuration System

- **Pydantic v2 models** for all configuration (6 sections: llm, project, spec, verification, export, security)
- **Layered merge**: hardcoded defaults < preset < `.intake.yaml` < CLI flags
- **3 presets**: `minimal` (cheap/fast prototyping), `standard` (balanced), `enterprise` (full traceability + risk assessment)
- **Budget enforcement**: `max_cost_per_spec` stops analysis if cost exceeds the configured limit
- **Auto-detection**: tech stack detected from 28+ project file markers + content patterns

### Spec Differ

Compare two spec versions with `intake diff`:

- Compares requirements by ID (FR-XX, NFR-XX)
- Compares tasks by number
- Compares acceptance checks by ID
- Reports added, removed, and modified items per section

### Doctor with Auto-Fix

`intake doctor --fix` automatically:

- Installs missing Python packages (detects `pip3.12`, `pip3`, or `pip`)
- Creates a default `.intake.yaml` configuration file

9 diagnostic checks: Python version, LLM API key, 6 optional dependencies, config file validity.

### Error Hardening

- `EmptySourceError` and `FileTooLargeError` with user-friendly messages and fix suggestions
- Centralized `validate_file_readable()` and `read_text_safe()` utilities
- UTF-8 to latin-1 encoding fallback
- 50 MB file size limit
- All 8 parsers use hardened validation

## Architecture

```
INGEST (parsers) --> ANALYZE (LLM) --> GENERATE (specs) --> VERIFY (checks) --> EXPORT (output)
```

- **Module isolation**: Only `analyze/` talks to the LLM. All other phases work offline.
- **Protocol over ABC**: `Parser`, `Exporter`, and `Reporter` use `typing.Protocol` with `@runtime_checkable`.
- **Dataclasses for pipeline data, Pydantic for config** — never mixed.
- **Async only in analyze/** — everything else is synchronous.
- **Custom exception hierarchy** per module with reason + suggestion messages.

## Quick Start

```bash
# Install
pip install intake-ai-cli

# Set your API key
export ANTHROPIC_API_KEY=sk-ant-...

# Check environment
intake doctor

# Generate a spec from a Markdown file
intake init "OAuth2 auth system" -s requirements.md

# Generate from multiple sources
intake init "Payments" -s jira.json -s confluence.html -s notes.md

# Verify implementation
intake verify specs/oauth2-auth-system/ -p .

# Export for an agent
intake export specs/oauth2-auth-system/ -f architect -o output/
```

## Known Limitations

- **PDF and DOCX parsers** have lower test coverage (15% and 14%) due to missing real binary fixtures. The code is complete and manually tested.
- **Image parser** ships with a stub — real LLM vision analysis requires configuring an LLM adapter with vision capability.
- **Export formats** `claude-code`, `cursor`, and `kiro` are declared in the CLI but not yet implemented. Use `architect` or `generic` for now.
- **`init` and `add` commands** require a live LLM connection. All other commands work fully offline.
