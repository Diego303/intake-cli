# Release v0.2.0 — Plugin System + New Parsers + Adaptive Generation

> Extensible architecture, 3 new parsers, complexity classification, and 492 tests.

This release transforms intake from a monolithic tool into a plugin-extensible platform. All parsers and exporters are now discoverable via Python entry_points (PEP 621), the generation pipeline adapts to source complexity, and task state tracking allows managing implementation progress directly in the spec.

## Install

```bash
pip install intake-ai-cli
```

Requires Python 3.12+.

- **Plugin system** with automatic discovery via entry_points (PEP 621)

## What's New

### Plugin System

intake now uses a plugin-first architecture. Parsers and exporters are discovered automatically via `importlib.metadata.entry_points()`.

```bash
# List all discovered plugins
intake plugins list

# Validate plugin compatibility
intake plugins check
```

**Three entry point groups:**

| Group | Content |
|-------|---------|
| `intake.parsers` | 11 built-in parsers |
| `intake.exporters` | 2 built-in exporters (V1) |
| `intake.connectors` | Infrastructure ready (no concrete connectors yet) |

**V2 Plugin protocols** extend V1 with metadata, confidence scoring, and additional capabilities:

- `ParserPlugin`: meta, supported_extensions, confidence(), can_parse(), parse()
- `ExporterPlugin`: meta, supported_agents, export() -> ExportResult
- `ConnectorPlugin`: meta, uri_schemes, can_handle(), fetch() (async), validate_config()

Existing V1 parsers continue to work without modification. Registries accept both V1 and V2.

**Pipeline hooks**: `HookManager` with register/emit pattern. Ready for event wiring in future releases.

### 3 New Parsers (11 Total)

| Parser | Format | What It Extracts |
|--------|--------|------------------|
| **URL** | `http://`, `https://` | Page content as Markdown via httpx + BS4 + markdownify. Auto-detects Confluence/Jira/GitHub from URL patterns. |
| **Slack** | `.json` (auto-detected) | Messages, threads, decisions (via reactions/keywords), action items. Groups by thread. |
| **GitHub Issues** | `.json` (auto-detected) | Issues, labels, assignees, milestones, comments, `#NNN` cross-references. Single or array format. |

JSON subtype detection order: Jira > GitHub Issues > Slack > generic YAML.

### Complexity Classification + Adaptive Generation

intake now automatically classifies source complexity and adjusts generation:

| Mode | Criteria | Files Generated |
|------|----------|-----------------|
| `quick` | <500 words, 1 source, no structure | `context.md` + `tasks.md` |
| `standard` | Default | All 6 spec files |
| `enterprise` | 4+ sources OR >5000 words | All 6 + detailed risks |

```bash
# Auto-detection (default when auto_mode is true)
intake init "Feature" -s reqs.md

# Force a mode
intake init "Quick fix" -s bug.txt --mode quick
intake init "Critical system" -s reqs.md --mode enterprise
```

### Task State Tracking

Track implementation progress directly in the spec:

```bash
# List tasks with status
intake task list specs/my-feature/

# Update task status
intake task update specs/my-feature/ 1 done --note "Implemented and tested"

# Filter by status
intake task list specs/my-feature/ --status pending --status in_progress
```

Statuses: `pending`, `in_progress`, `done`, `blocked`. Changes are persisted in `tasks.md`.

### Source URI Resolution

The `init` command now resolves source URIs:

| Input | Resolution |
|-------|-----------|
| `reqs.md` | Local file -> parser registry |
| `https://wiki.com/page` | URL -> UrlParser |
| `-` | Stdin -> PlaintextParser |
| `jira://PROJ-123` | Scheme URI -> displays "connector not available yet" warning |
| Free text | Plaintext fallback |

### Configuration Updates

- `spec.auto_mode: bool = True` — enables automatic complexity classification
- `ConnectorsConfig` with sub-models for Jira, Confluence, GitHub (preparation for v0.3.0 connectors)

| Command | Description |
|---------|-------------|
| `intake plugins list` | List all discovered plugins with name, group, version, V2 status |

## Architecture Changes

### New Modules

| Module | Purpose |
|--------|---------|
| `plugins/protocols.py` | V2 protocols: ParserPlugin, ExporterPlugin, ConnectorPlugin |
| `plugins/discovery.py` | Entry point scanning via importlib.metadata |
| `plugins/hooks.py` | Pipeline hook system (HookManager) |
| `connectors/base.py` | ConnectorRegistry, ConnectorError (infrastructure, no concrete connectors) |
| `ingest/url.py` | HTTP/HTTPS URL parser |
| `ingest/slack.py` | Slack workspace export parser |
| `ingest/github_issues.py` | GitHub Issues JSON parser |
| `analyze/complexity.py` | Heuristic complexity classification |
| `generate/adaptive.py` | Mode-aware spec file selection |
| `utils/source_uri.py` | URI parsing (file, url, stdin, scheme) |
| `utils/task_state.py` | Task status management in tasks.md |

### Modified Modules

| Module | Change |
|--------|--------|
| `ingest/registry.py` | Plugin discovery + JSON subtype detection for Slack/GitHub |
| `export/registry.py` | Plugin discovery for exporters |
| `analyze/models.py` | `TaskItem.status` field (default: "pending") |
| `templates/tasks.md.j2` | Status column in summary table |
| `config/schema.py` | `SpecConfig.auto_mode`, `ConnectorsConfig` sub-models |
| `cli.py` | plugins/task groups, init --mode, source URI resolution |

## Migration Guide

No migration needed. All existing specs, configurations, and workflows continue to work unchanged.

To use the new features:

```bash
# Upgrade
pip install --upgrade intake-ai-cli

# Verify
intake --version     # 0.2.0

# Try new features
intake init "Feature" -s https://wiki.com/page      # URL source
intake init "Quick fix" -s notes.txt --mode quick    # quick mode
intake task list specs/my-feature/                    # task tracking
```
