# Release v0.6.0 — GitLab, Validate, Estimate, Custom Templates, CI Export

> GitLab API connector + parser, offline spec validation, LLM cost estimation, customizable templates, and CI config generation.

This release completes Phase 5 of intake's roadmap: a GitLab API connector with full issue/milestone support, a GitLab Issues parser, an offline spec validator (`intake validate`), an LLM cost estimator (`intake estimate`), a Jinja2 template override system for customizing generated output, CI config generation (`intake export-ci`) for GitLab CI and GitHub Actions, and 2 new MCP tools. 107 new tests bring the total to 882.

## Install

```bash
pip install intake-ai-cli

# With all connector dependencies (Jira, Confluence, GitHub, GitLab)
pip install "intake-ai-cli[connectors]"

# Everything
pip install "intake-ai-cli[all]"
```

Requires Python 3.12+.

## What's New

### GitLab API Connector

Pull issues directly from GitLab without manual export:

```bash
# Single issue
intake init "Bug fix" -s gitlab://org/repo/issues/42

# Multiple issues
intake init "Sprint" -s gitlab://org/repo/issues/42,43,44

# Filtered query
intake init "Backend bugs" -s "gitlab://org/repo/issues?labels=bug&state=opened"

# Milestone issues
intake init "v2.0 scope" -s gitlab://org/repo/milestones/3/issues

# Nested groups (GitLab-specific)
intake init "Feature" -s gitlab://org/team/subgroup/project/issues/10
```

Features:
- **python-gitlab v8.x** with lazy import — no startup cost if not used
- **Nested group support** — URI parsing handles any group depth
- **SSL config** — `ssl_verify: false` for self-hosted instances with custom certs
- **Notes, MRs, labels** — configurable extraction depth
- **Async fetch** — consistent with other connectors

#### Configuration

```yaml
# .intake.yaml
connectors:
  gitlab:
    url: "https://gitlab.company.com"     # Default: https://gitlab.com
    token_env: GITLAB_TOKEN               # env var with API token
    auth_type: token                      # token | oauth
    default_project: "team/backend"       # Optional default project path
    include_comments: true                # Include issue notes
    include_merge_requests: true          # Include related MRs
    max_notes: 10                         # Max notes per issue
    ssl_verify: true                      # false for self-signed certs
```

### GitLab Issues Parser

Auto-detects GitLab Issues JSON by the presence of the `iid` field. Supports three input formats:

| Format | Example |
|--------|---------|
| Single issue | `{"iid": 42, "title": "...", ...}` |
| Array | `[{"iid": 42, ...}, {"iid": 43, ...}]` |
| Wrapped | `{"issues": [{"iid": 42, ...}]}` |

Extracts:
- Title, description, state, labels, milestone
- Notes/comments (truncated to 500 chars each)
- Related merge requests
- Metadata: `issue_count`, `label_list`, `has_notes`, `has_merge_requests`

JSON auto-detection priority: Jira > GitLab Issues > GitHub Issues > Slack > generic YAML.

### Spec Validator (`intake validate`)

Offline quality gate that checks spec consistency without calling any LLM:

```bash
# Standard validation
intake validate specs/my-feature/

# Strict mode (warnings also fail)
intake validate specs/my-feature/ --strict

# JSON output for CI
intake validate specs/my-feature/ --format json
```

#### 5 Check Categories

| Category | What It Checks |
|----------|---------------|
| `structure` | Required files exist (`requirements.md`, `tasks.md`, etc.) |
| `cross_reference` | Task→requirement references are valid, no dangling refs |
| `consistency` | No cycles in task dependency graph (DFS detection) |
| `acceptance` | Checks in `acceptance.yaml` have required fields |
| `completeness` | Every requirement has at least one implementing task |

Exit codes: `0` = valid, `1` = issues found, `2` = execution error.

### Cost Estimator (`intake estimate`)

Estimate LLM costs before running `intake init`:

```bash
# Estimate from source files
intake estimate -s requirements.md -s notas.md

# Specify model
intake estimate -s reqs.md --model gpt-4o

# Specify mode
intake estimate -s reqs.md --mode enterprise

# JSON output
intake estimate -s reqs.md --format json
```

#### 7-Model Pricing Table

| Model | Input $/1M tokens | Output $/1M tokens |
|-------|-------------------|-------------------|
| claude-sonnet-4 | $3.00 | $15.00 |
| claude-opus-4 | $15.00 | $75.00 |
| gpt-4o | $2.50 | $10.00 |
| gpt-4o-mini | $0.15 | $0.60 |
| gpt-4-turbo | $10.00 | $30.00 |
| gemini-2.0-flash | $0.10 | $0.40 |
| gemini-2.5-pro | $1.25 | $10.00 |

Unknown models fall back to claude-sonnet-4 pricing. Budget warnings are emitted when the estimate exceeds `llm.max_cost_per_spec`.

### Custom Template Loading

Override any of intake's 17 built-in Jinja2 templates by placing a file with the same name in `.intake/templates/`:

```
my-project/
├── .intake/
│   └── templates/
│       └── requirements.md.j2   # Your custom format
└── .intake.yaml
```

The `TemplateLoader` uses Jinja2 `ChoiceLoader`:
1. User templates in `.intake/templates/` (highest priority)
2. Built-in templates in the intake package (fallback)

Override detection logs an `INFO` message when a built-in template is shadowed.

```yaml
# .intake.yaml
templates:
  user_dir: ".intake/templates"   # Default
  warn_on_override: true          # Log when overrides are detected
```

### CI Export (`intake export-ci`)

Generate CI configuration that runs `intake validate` and `intake verify` on every PR:

```bash
# GitLab CI
intake export-ci specs/my-feature/ --platform gitlab

# GitHub Actions
intake export-ci specs/my-feature/ --platform github

# Custom output path
intake export-ci specs/my-feature/ --platform gitlab --output ci/
```

| Platform | Generated File | Contents |
|----------|---------------|----------|
| GitLab | `.gitlab-ci.yml` | `verify` stage with intake validate + verify, JUnit artifacts |
| GitHub | `.github/workflows/intake-verify.yml` | Job with checkout, setup-python, validate + verify, artifact upload |

Both templates are customizable via the template override system.

### Technical Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 39 | python-gitlab lazy import | No startup cost; clear error message when not installed |
| 40 | GitLab nested groups via `/issues/` keyword | Supports `org/team/subgroup/project` without path depth limits |
| 41 | Validator 100% offline | No LLM dependency — `validate/` never imports from `llm/` or `analyze/` |
| 42 | DFS cycle detection in tasks | `visited` + `in_stack` sets detect cycles and report the full cycle path |
| 43 | ValidateConfig `alias="validate"` | `validate` is a Pydantic BaseModel method — alias avoids shadowing |
| 44 | 7-model pricing with fallback | Unknown models use claude-sonnet-4 pricing as reasonable default |
| 45 | Template ChoiceLoader | User FileSystemLoader takes priority over built-in PackageLoader |
| 46 | MCP estimate scans spec files | Estimates from `.md`/`.yaml` in spec dir without needing original sources |

## Migration Guide

No migration needed.

```bash
# Upgrade
pip install --upgrade intake-ai-cli

# Install GitLab connector support (optional)
pip install "intake-ai-cli[connectors]"

# Verify
intake --version     # 0.6.0

# Try GitLab connector
export GITLAB_TOKEN=glpat-your-token
intake init "Sprint" -s "gitlab://org/repo/issues?labels=sprint"

# Try spec validation
intake validate specs/my-feature/

# Try cost estimation
intake estimate -s requirements.md

# Try CI export
intake export-ci specs/my-feature/ --platform github

# Try custom templates
mkdir -p .intake/templates
cp path/to/intake/templates/requirements.md.j2 .intake/templates/
# Edit .intake/templates/requirements.md.j2 to customize
intake init "Test" -s reqs.md  # Uses your custom template
```
